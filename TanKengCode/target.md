6.4 基于预测结果的附加调度应用演示
6.4.1 基于洪水预报的预泄触发与预警演示

这一节的目标是展示：
大模型拿到预测结果后，如何判断“是否需要预泄、预泄到什么目标、是否需要同步触发预警”。

这一节最适合承接 6.3 的输出，即：

当前库水位
当前入库流量
未来若干小时或若干天入库流量预测过程
当前所属汛期阶段（梅汛期、台汛期、过渡期）
当前下游约束条件

控制运用计划中已经给了这一场景最关键的规则依据：
梅汛期、非汛期控制蓄水位 160.00m，台汛期控制蓄水位 156.50m；梅台过渡期若有台风、暴雨影响，必须及时降至 156.50m；当预报有大暴雨或洪水、且会商分析认为库水位将超过控制蓄水位时，应通过预泄适当降低库水位。计划还要求，当库水位逼近汛限水位时，应坚持“早动、快动、小动”，及时汇报并实施主动性预泄。

这一节建议按下面的步骤来组织：

步骤1：读取预测结果与实时状态
调用实时状态类 MCP 服务，获取当前水位、入库流量、出库流量、降雨、库容；
调用预测类 MCP 服务，获取未来时段入库流量预测过程、峰值、峰现时间、可能最高库水位。
这一步的产物是“调度判断输入包”。

步骤2：识别当前运行阶段
调用规则识别/日历判断服务，确定当前属于梅汛期、台汛期还是过渡期。
这里的判断直接影响控制水位阈值：梅汛期 160.00m，台汛期 156.50m。

步骤3：执行预泄触发判定
调用规则判定 MCP 服务，输入：

当前库水位
预测最高库水位
当前时段类型
未来天气/台风影响标识

输出：

是否触发预泄
触发原因
目标控制水位
是否需要同步预警

这里建议把判定逻辑写成显式规则：

若预测最高库水位不超过当前阶段控制蓄水位，则“不触发预泄”
若预测最高库水位超过控制蓄水位，则“触发预泄”
若处于梅台过渡期，且预报有台风/暴雨影响，则“必须降至 156.50m”
这些都能直接从计划中取依据。

步骤4：生成预泄目标与预警建议
调用调度建议生成 MCP 服务，输出：

建议预泄目标水位
建议预泄开始时间
建议预泄时长
是否需要触发高水位预警
是否需要触发泄洪放水预警

计划中明确了两类预警：
一类是高水位预警，当蓄水位超过 160.00m 且预报最高水位将达到 161.50m 时，应报告并发布高水位预警；
另一类是泄洪放水预警，当水位逼近汛限水位（梅汛期 159.50m、台汛期 156.00m）且预计有弃水可能时，应及时汇报并实施主动性预泄，预定闸门开启前 1 小时拉泄洪警报。

步骤5：输出可执行结果
结果输出建议包括：

是否预泄
目标控制水位
触发依据
预警建议
下一步进入“方案生成”还是“继续监测”

这一节的MCP 服务建议分工可以写成：

状态读取服务
预测结果读取服务
汛期阶段识别服务
规则判定服务
预警建议生成服务
留痕与日志服务
6.4.2 基于预测结果的下泄方案生成与比选演示

这一节的目标是展示：
在预泄已经触发或洪水预报已经形成之后，大模型如何生成候选下泄方案，并依据规则和模拟结果进行比选。

这部分最重要，因为控制运用计划已经把水位区间和调度规则写得非常细。
例如：

水位在控制蓄水位至 161.50m 时，按补偿凑泄方式调度，并控制鹤城站不超过 14000m3/s
若预报区间洪水达到或超过 14000m3/s，则水库关闸
水位超过 161.5m 后逐渐开启溢洪道闸门
水位在 161.5m～161.7m 时按 6000m3/s 控制下泄
水位超过 161.7m 时溢洪道闸门全开、机组参与泄洪
水位超过 165.27m 时，溢洪道闸门全开并开启泄洪洞，发电暂停。

此外，计划还给了：

溢洪道和泄洪洞泄流能力表/曲线
台汛期、梅汛期防洪特征水位和最大泄流量成果表
闸门开启顺序：#1 → #6 → #3 → #4 → #2 → #5
这些都非常适合直接喂给 MCP 调度服务做方案计算。

这一节建议按下面步骤组织：

步骤1：汇总调度决策输入
调用以下 MCP 服务汇总输入：

预测过程读取服务：未来来水过程、峰值、峰现时间
实时状态服务：当前库水位、库容、出入库流量
下游约束服务：鹤城站流量约束
能力曲线服务：溢洪道泄流能力、泄洪洞泄流能力、水位-库容关系
调度规则服务：不同水位区间的操作规则

这一步产出统一的“方案生成输入包”。

步骤2：进行调度规则分区判断
调用规则分区服务，根据当前或预测最高库水位判断落入哪个规则区间：

控制蓄水位以内
控制蓄水位～161.50m
161.50m～161.70m
161.70m～165.27m

165.27m

该步骤只负责“定区间”，不直接给方案。规则依据来自控制运用计划第 2.5 节。

步骤3：生成候选下泄方案
调用方案生成服务，至少输出 2～3 个候选方案，例如：

方案A：保守型，优先削峰，较早启动预泄
方案B：均衡型，兼顾防洪与发电
方案C：紧约束型，严格按最小必要泄量控制

每个方案至少应包含：

起调时刻
目标下泄流量过程
机组是否参与
是否使用溢洪道
是否启用泄洪洞
闸门组合与开度安排
对应的预期最高库水位

这里要特别加一个子步骤：
根据闸门规则生成闸门操作序列，并按计划中的开启顺序和关闭顺序生成可执行闸门方案。

步骤4：调用调洪/水库演算服务进行模拟
这是最关键的 MCP 调用环节。
每个候选方案都要调用一次调洪演算/水库路由模拟服务，输出至少包括：

最高库水位
最大下泄流量
各时段出库过程
是否超过鹤城站流量约束
是否进入更高风险规则区间
是否触发发电暂停
弃水量/发电损失等附加指标

这一节可以直接复用计划中的调洪成果表、泄流能力表和防洪调度图作为规则基础与结果校验依据。

步骤5：执行方案比选与推荐
调用方案评分/比选服务，对多个候选方案进行比较。
建议设置的评分维度包括：

防洪安全性：是否控制库水位不过关键阈值
下游安全性：是否满足鹤城站流量控制
规则符合性：是否严格遵守计划规定
调度平稳性：闸门动作是否过于剧烈
兴利影响：发电影响和弃水程度

最终输出：

推荐方案
不推荐其他方案的原因
方案适用前提
关键风险提示

这一节的MCP 服务建议分工可以写成：

规则分区判定服务
泄流能力查询服务
闸门方案生成服务
调洪演算服务
方案评分服务
结果汇总与解释服务
6.4.3 洪水过程中的动态调度指令更新演示

这一节的目标是展示：
洪水过程不是一次性决策，大模型如何在新数据、新预报或新指令到来后，对原有调度方案进行动态更新。

这部分和你前面的“动态更新预测”正好呼应。
计划中已经明确要求强化“四预”措施，及时处理异常情况，严格执行控制运用计划和预警方案；同时在泄洪预警部分也强调，当逼近汛限水位且预计有弃水可能时，应及时汇报并主动预泄。

这一节建议按下面步骤组织：

步骤1：建立“原始方案 + 新输入”的更新触发机制
系统需要持续监听以下更新源：

新降雨预报
新水位/流量观测
新洪水预报结果
新调度指令
新的下游约束信息

调用“更新触发判定服务”，判断是否满足重算条件。
典型触发条件可写成：

预测峰值显著变化
预测最高库水位跨越关键阈值
当前执行方案将不再满足控制运用要求
调度权限发生变化
外部指令发生变化

步骤2：重新调用预测与规则识别服务
一旦触发更新，系统应先调用更新后的预测服务，再重新判定：

当前规则区间
当前调度权限
当前是否需要预警
原方案是否仍然有效

这里不需要重新从零规划整章，而是对现有方案做“续算 + 纠偏”。

步骤3：生成“增量式调度指令”
这是本节最重要的输出方式。
不要只输出一个新的完整方案，还要输出：

与原方案相比的变化项
哪些闸门操作需要修改
哪些时段下泄过程需要调整
是否需要提前/延后/加大/减小下泄
是否需要切换至更高等级运行方式

也就是说，这里更适合调用“指令更新服务”而不是单纯“方案生成服务”。

步骤4：触发预警与通知联动
如果新方案涉及：

开闸时刻变化
流量级别明显变化
高水位预警触发
下游风险变化

则应同步调用预警推送/通知服务。
计划中明确要求预定闸门开启前 1 小时拉泄洪警报并对下游巡查，因此这个步骤在场景设计里要保留。

步骤5：记录版本、保留留痕
每次动态调整都应留痕，包括：

更新时间
更新原因
原方案摘要
新方案摘要
差异说明
触发依据

这样写很适合后面做“调度指令更新记录”展示。

这一节的MCP 服务建议分工可以写成：

实时监测更新服务
预测重算服务
规则重判服务
增量调度指令生成服务
预警通知服务
版本管理/日志留痕服务
6.4.4 通讯孤岛与异常工况下的应急调度演示

这一节我建议你保留。
原因是控制运用计划中“通讯孤岛”部分非常具体，而且它特别适合展示大模型系统在异常工况下如何从“辅助分析”转向“应急规则执行支撑”。这会让 6.4 节更完整，也更贴近真实水库运行。

这一节的目标是展示：
在与主管部门通讯异常或部分系统异常时，系统如何依据既定规则给出本地应急调度建议。

建议按下面步骤组织：

步骤1：识别异常工况
调用异常状态检测服务，识别：

与水利局通信中断
上级调度令无法及时获取
部分外部链路不可用

步骤2：执行联络尝试流程
调用应急联络服务，按计划要求依次尝试：

卫星电话
紧水滩电站中转联系
人工汇报
这一步体现“先联系、后本地应急执行”的规则。

步骤3：若仍无调度令，则按水位分区执行本地应急规则判定
调用“孤岛规则判定服务”，根据当前或预测库水位进入不同规则：

<161.50m：若无调度令，除发电流量外不泄洪
161.50m～162.84m：下泄不超过 8630m3/s
162.84m～163.54m：下泄不超过 9270m3/s
163.54m～164.30m：下泄不超过 9970m3/s
164.30m～165.27m：下泄不超过 10900m3/s

165.27m：溢洪道闸门全开并开启泄洪洞，发电暂停，不再控制下泄流量
这些规则在计划中写得很明确，非常适合做规则引擎演示。

步骤4：生成本地应急调度建议
输出内容建议包括：

当前处置级别
推荐开闸方式
推荐下泄上限
是否暂停发电
何时切换到下一阶段规则
恢复通信后如何交接

步骤5：恢复通信后的状态回传
系统应在通信恢复后自动整理：

孤岛期间执行了哪些操作
每一步依据是什么
当前库水位、出库流量、闸门状态是什么
是否需要补发报文或上报留痕

这一节的MCP 服务建议分工可以写成：

异常检测服务
应急联络服务
孤岛规则判定服务
本地调度建议服务
恢复上报服务
6.4 节整体建议的“步骤模板”

为了方便你后面直接接 MCP，我建议 6.4 每个场景统一采用同一套写法：

场景目标
这一场景要展示什么能力。

输入
当前状态、预测结果、计划规则、实时更新、外部约束。

步骤1：任务识别
大模型判断这是“预泄”“下泄方案”“动态更新”还是“应急调度”。

步骤2：规则读取
调用计划规则检索服务，拉取对应条款、阈值和流程。

步骤3：能力调用
按需要调用预测结果服务、状态查询服务、泄流能力服务、调洪演算服务等。

步骤4：生成候选结果
形成预泄建议、下泄方案、增量指令或应急调度建议。

步骤5：模拟/校核
检查是否符合控制水位、下游安全、权限和规则要求。

步骤6：结果输出
输出结构化结果、文本解释、预警建议和操作建议。

步骤7：留痕与回传
记录本次判断依据、调用过程、输出结果和指令版本。

最后一个建议

6.4 的正文不要写成“介绍调度理论”，要写成演示链路。
也就是每个三级标题下，都尽量让人一眼看到：

输入是什么 → 系统调用了什么 → 判定了什么 → 输出了什么

这样你后面最容易把已经写好的 MCP 服务直接挂进去。
