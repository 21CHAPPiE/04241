# 确认关口表单

## 一、确认关口总述

### 1. 关口目的

本次确认关口的目的，是把前序“项目评估 + 场景设计”阶段中与方案方向、第一版边界、验收倾向、交付形态有关的非技术事项收口，并判断当前是否已经具备进入方案定型的条件。

之所以现在不能直接跳到方案定型，是因为前序材料里虽然已经明确了基础闭环、第一版场景取舍和验收倾向，但甲方侧关于调度对象边界、交付形态、验收口径等仍有部分未明确内容；需要先判断这些未明确项是否仍属于阻断项，还是可以按通用默认值先进入下一阶段。前置项目分析中原本也将“调度对象边界、验收口径、交付形态”等列为需确认事项。

### 2. 输入来源

当前表单基于以下输入形成：

1. 第一阶段项目评估稿，对项目问题定义、最低实现边界、待确认事项和近期推进建议作了约束。

2. 第二阶段场景设计结果，已将业务流程收敛为“多模型预测筛选 → 贝叶斯加权 → 概率预报 → 调度方案生成”的基础闭环，并补充了两个异常场景。

3. 你刚刚明确给出的历史确认内容，包括第一版只做单目标、两个异常场景都进入第一版、偏向功能验收、第一版必须覆盖完整联动链路、异常场景按伪数据构造等。

### 3. 进入条件

满足以下条件时，可以进入方案定型：

1. 所有 P1 项均已明确为已确认。

2. 不再存在会改变项目问题定义、第一版边界、基础场景范围、验收倾向或交付形态的非技术未决项。

3. 剩余未决事项仅限于可按默认值暂时带过的低优先级事项，或属于后续技术细化问题。

## 四、关口判断结果

### 1. 已满足条件

1. 项目第一版方向已明确为“预测驱动调度闭环”，不存在根本性摇摆。

2. 第一版基础场景范围已明确，且主链路已经固定。

3. 验收倾向已明确为功能验收。

4. 第一版联动要求已明确为必须覆盖完整链路。

5. 异常场景是否进入第一版已经明确。

6. 剩余未明确内容中，大部分已可按“单库原型 + 原型交付”的通用默认值先行收口，不影响进入方案定型。

### 2. 未满足条件

1. 甲方侧尚未正式明确调度对象边界和最终交付形态，但目前可按默认待审方式暂时承接。

2. 验收口径的细化指标尚未明确，但该问题更接近方案定型和实施准备层面的细化，不再构成当前阻断项。前置项目分析中虽列出验收口径待确认，但当前“偏功能验收 + 完整闭环演示”的方向已经足以支撑下一阶段。

### 3. 当前判断

**可以进入方案定型**

当前所有真正会改变问题定义、第一版边界、基础场景范围和验收倾向的 P1 核心事项，已经确认或已按稳妥默认值收口；剩余未决内容主要是细化口径问题，不再阻断进入下一阶段。

## 五、下一步动作建议

### 1. 需优先完成的确认动作

1. 在方案定型稿首页直接显式写明：第一版按“单库原型 + 单目标调度 + 完整联动链路 + 两个异常修正场景”推进。

2. 在方案定型稿中单独设一节写清“默认待审项”，把“单库边界”和“原型交付形态”固化下来，避免后续反复摇摆。

3. 把“已确认的历史事项”转写为方案定型稿的输入约束，而不是重新讨论。

### 2. 进入方案定型前的收口建议

建议只再做一次最小收口，不再扩大确认范围。重点是把当前已经确认的内容整理成三类：

第一类，必须写入方案边界的硬约束。  
第二类，可以默认采用但需标注待审的边界项。  
第三类，后续技术设计时再展开的细化项。

这样可以避免在方案定型阶段重新混入非技术讨论。

### 3. 进入方案定型后的衔接建议

进入方案定型后，建议重点处理以下三件事：

第一，把基础闭环拆成正式功能链路：多模型接入与筛选、贝叶斯加权概率预报、单目标调度求解、异常修正触发。

第二，把两个异常场景写成“触发条件—重算范围—结果输出”的标准化处理链，而不是单独散写。

第三，把配置化要求单列为横向能力，包括阈值配置、权重配置、目标配置和规则配置，作为后续方案结构的统一约束。
