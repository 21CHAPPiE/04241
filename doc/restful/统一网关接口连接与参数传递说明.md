# 统一网关接口连接与参数传递说明

## 1. 文档目的

本文档用于说明当前项目在统一网关模式下的接口连接方式，重点回答以下问题：

1. 外部系统应该连接哪个服务入口
2. `HuadongCode` 预测链路应该如何串接
3. `TanKengCode` 调度链路应该如何串接
4. 每一步的输入和输出分别是什么
5. 下一个模块如何拿到上一个模块的结果
6. 路径型参数是否应该改成数组，或者使用其他传参方式

本文档以当前仓库中的真实代码为准：

- 统一网关入口：[UnifiedGateway/gateway/app.py](/H:/04241/UnifiedGateway/gateway/app.py)
- 华东 REST 入口：[HuadongCode/app/rest_api.py](/H:/04241/HuadongCode/app/rest_api.py)
- 滩坑 REST 入口：[TanKengCode/tanken_rest_api.py](/H:/04241/TanKengCode/tanken_rest_api.py)

---

## 2. 总体结论

当前项目建议按如下方式对接：

- 对外统一连接 `UnifiedGateway`
- 做预测走 `/huadong/...`
- 做调度走 `/tanken/...`
- 耗时任务统一走 `POST .../jobs` + `GET .../jobs/{job_id}`

默认网关地址：

```text
http://localhost:8010
```

常用健康检查：

- `GET /health`
- `GET /huadong/health`
- `GET /tanken/health`

也就是说，外部系统只需要记住一个 HTTP 入口，再根据路径前缀访问不同业务域。

---

## 3. 统一网关与两个子服务的关系

统一网关本身不负责水文计算或调度计算，它只是将两个子服务挂载到一个统一的 FastAPI 服务中。

挂载关系如下：

- `HuadongCode` 挂载在 `/huadong`
- `TanKengCode` 挂载在 `/tanken`

因此实际访问规则如下：

- 华东水文预测接口：
  - `/huadong/...`
- 滩坑调度接口：
  - `/tanken/...`

建议外部系统始终走统一网关，而不是直接分别连接两个子项目。这样做的优点是：

- 对外只有一个服务地址
- 便于反向代理、鉴权、日志收集和运维治理
- 华东和滩坑的接口前缀清晰，不容易混淆

## 4. Huadong 预测链路如何连接

## 4.1 推荐主链

推荐按以下顺序连接：

1. `POST /huadong/dataset/profile`
2. `POST /huadong/model-assets/profile`
3. `POST /huadong/forecast`
4. `POST /huadong/ensemble`
5. `POST /huadong/correction`

如果需要进一步分析和业务输出，可继续接：

6. `POST /huadong/analysis`
7. `POST /huadong/risk`
8. `POST /huadong/warning`

其中最核心的生产链是：

```text
profile -> assets -> forecast -> ensemble -> correction
```

---

## 4.2 Huadong 请求模型说明

从代码看，华东侧主要使用两种请求结构：

### 1. 路径型请求

```json
{
  "dataset_path": "string | null",
  "file_path": "string | null",
  "output_root": "string | null",
  "options": {}
}
```

用途：

- `dataset_path`
  - 传原始输入数据文件
- `file_path`
  - 传上一步已经生成的产物文件
- `output_root`
  - 指定当前步骤产物输出目录
- `options`
  - 额外业务参数

### 2. 仅输出型请求

```json
{
  "output_root": "string | null",
  "options": {}
}
```

用途：

- 常用于模型资产概况读取，不需要输入数据文件

## 4.4 Huadong 上下游结果如何衔接

华东链路的核心原则是：

```text
每个步骤都从上一步返回的 artifact_paths 中拿到真实文件路径，再传给下一步的 file_path。
```

标准衔接方式如下：

1. `forecast` 返回：

```json
{
  "artifact_paths": {
    "forecast": "实际生成的 forecast.csv"
  }
}
```

2. `ensemble` 请求中写：

```json
{
  "file_path": "<forecast.artifact_paths.forecast>"
}
```

3. `ensemble` 返回：

```json
{
  "artifact_paths": {
    "ensemble": "实际生成的 ensemble.csv"
  }
}
```

4. `correction` 请求中写：

```json
{
  "file_path": "<ensemble.artifact_paths.ensemble>"
}
```

5. `correction` 返回：

```json
{
  "artifact_paths": {
    "correction": "实际生成的 corrected.csv"
  }
}
```

6. `risk` / `warning` 请求中写：

```json
{
  "file_path": "<correction.artifact_paths.correction>"
}
```

注意：

- 文档中像 `"H:\\0424\\doc\\restful\\outputs\\huadong\\correction\\...\\corrected.csv"` 这种写法只是示意
- 其中的 `...` 不是字面值，不能原样传
- 正确做法是从实际响应里读取完整路径

## 5.3 Tanken 上下游结果如何衔接

滩坑链路不是“文件产物串文件”，而是“决策结果串业务参数”。

最关键的衔接是：

1. `optimize` 返回：

```json
{
  "selected_module_type": "constant_release",
  "selected_module_parameters": {
    "target_release": 2283.657
```

2. `simulate` 请求中写：

```json
{
  "target_outflow": "<optimize.avg_release_m3s>",
  "module_type": "<optimize.selected_module_type>",
  "module_parameters": "<optimize.selected_module_parameters>"
}
```

3. `evaluate` 通常复用同一组参数：

```json
{
  "target_outflow": "<optimize.avg_release_m3s>",
  "module_type": "<optimize.selected_module_type>",
  "module_parameters": "<optimize.selected_module_parameters>"
}
```

所以 Tanken 的主规则是：

```text
用 optimize 的输出字段直接作为 simulate / evaluate 的输入字段。
```

## 7. 路径参数是否应该改成数组

## 7.1 当前接口的真实情况

当前代码中，华东侧 `dataset_path` 和 `file_path` 都是单个字符串：

```json
{
  "dataset_path": "string | null",
  "file_path": "string | null"
}
```

也就是说：

- 当前接口设计本身不支持数组
- 也没有定义 `files: []` 或 `paths: []` 这类结构

因此，如果直接把：

```json
"file_path": "H:\\04241\\doc\\restful\\outputs\\huadong\\correction\\...\\corrected.csv"
```

改成：

```json
"file_path": [
  "path1",
  "path2"
]
```

在当前实现下是不兼容的。

---

## 7.2 数组传参是否合适

结论：

- 对当前这套接口而言，**不推荐直接把 `file_path` 改成数组**。

原因如下：

### 1. 当前每个接口语义上只消费一个主输入文件

例如：

- `ensemble` 的输入就是一个 `forecast.csv`
- `correction` 的输入就是一个 `ensemble.csv`
- `risk` / `warning` 的输入就是一个 `corrected.csv`

如果把 `file_path` 直接改成数组，会出现两个问题：

- 这个接口到底取数组中的哪个文件
- 多个文件的顺序是否有语义

这样会让接口契约变得模糊。

### 2. 数组不利于接口校验

如果只是传：

```json
"file_path": ["a.csv", "b.csv"]
```

接口端还需要额外判断：

- 必须传几个文件
- 每个文件分别代表什么角色
- 顺序错了怎么办

这会增加不必要的歧义。

### 3. 当前链路更适合“单步单输入、显式衔接”

当前这套设计的优势就是：

- 上一步输出什么
- 下一步明确吃哪个产物

链路非常清晰，适合工程联调和排障。

---

## 7.3 更合理的替代方案

如果未来确实需要“多个上游产物一起传”，推荐不要用裸数组，而是改成具名对象。

### 推荐方案 A：继续保持单路径设计

这是当前最推荐的方式。

优点：

- 最清晰
- 与现有代码完全兼容
- 最容易定位问题

### 推荐方案 B：使用具名输入对象，而不是数组

例如未来扩展时，可以设计成：

```json
{
  "input_artifacts": {
    "forecast_csv": "xxx",
    "observation_csv": "yyy",
    "weights_json": "zzz"
  },
  "output_root": "..."
}
```

优点：

- 每个输入文件的语义明确
- 不依赖数组顺序
- 更容易扩展和校验

### 推荐方案 C：单独增加 batch 接口

如果未来业务确实需要批量处理多个文件，建议新开批量接口，而不是重载现有单文件接口。

例如：

```text
POST /huadong/ensemble/batch
POST /huadong/correction/batch
```

这样可以把：

- 单文件处理接口
- 多文件批处理接口

明确区分开。

---

## 7.4 对当前项目的最终建议

对当前项目，建议如下：

1. 保持现有 `dataset_path` / `file_path` 单字符串设计不变
2. 每一步都从上一步的 `artifact_paths` 里读取真实路径
3. 不要使用带 `...` 的示意路径作为真实参数
4. 如果后续确实要支持多输入，优先考虑“具名对象”而不是“裸数组”
5. 如果后续确实要支持批量任务，建议新开 batch 接口，而不是修改现有接口含义

## 8. 两条最推荐的连接方式

## 8.1 华东预测对接方式

推荐最小生产链：

```text
/huadong/dataset/profile

-> /huadong/forecast
-> /huadong/ensemble
```

- `ensemble.file_path = forecast.artifact_paths.forecast`

如果需要扩展：

- `risk.file_path = correction.artifact_paths.correction`
- `warning.file_path = correction.artifact_paths.correction`

## 8.2 滩坑调度对接方式

推荐最小生产链：

```text
/tanken/cases

-> /tanken/cases/{case_id}/rules
-> /tanken/cases/{case_id}/optimize
-> /tanken/cases/{case_id}/simulate
-> /tanken/cases/{case_id}/evaluate
```

连接规则：

- `simulate.target_outflow = optimize.avg_release_m3s`
- `simulate.module_type = optimize.selected_module_type`
- `simulate.module_parameters = optimize.selected_module_parameters`
- `evaluate` 通常复用同一组参数

## 9. 最终结论

当前项目的对接思想非常明确：

- 网关统一入口
- 华东走文件型流水线
- 滩坑走场景型决策链
- 异步任务统一走 `jobs`

对于“路径参数要不要改成数组”这个问题，当前答案是：

```text
不建议。
```

更稳妥的做法是：

- 当前版本继续保持单路径串接
- 上一步从 `artifact_paths` 输出真实产物路径
- 下一步显式把该路径传入 `file_path`

这也是当前代码最清晰、最稳定、最适合工程联调的用法。
